一、这个项目到底在做什么
你这个项目的真实目标，不是“让 Jetson 也能跑 3DGS”，而是解决一个更深的问题：
在边缘设备上，在线 3D 地图怎样在长期运行时不崩。

===== Page 2 =====
因为长期在线建图同时有四种互相冲突的需求：
1.
局部区域要高保真
2.
历史信息要长期保留
3.
显存和功耗必须受控
4.
重访时历史区域还要能被找回并恢复
如果你把这四件事都压在一个不断膨胀的 online state 里，不管它是 recurrent state 还是
global Gaussian map，最后都会炸：地图越来越大、更新越来越重、旧信息越来越不稳
定、重访恢复越来越差。你报告里对这一点的判断是对的。
所以你提出的解法是三层状态：
M_short
：短时观测和描述子
M_active
：当前活动高斯子图，常驻 GPU
B_bank
：归档历史子图，放在 CPU/NVMe
再配一个生命周期：
observe
promote
archive
retrieve
recover
这就把“建图”从一个单一持续优化对象，变成一个
会管理记忆生命周期的系统

四、真正会决定成败的，不是“高斯能不能跑”，而是这三个点
第一，retrieve/recover 的正确性
这是你系统最像“研究”的部分，也是最难的部分。
因为如果 retrieve 出错，系统会把错误历史注入当前 active map，结果不是“没增益”，而
是
主动污染地图
。你报告里已经写了要做 descriptor recall + geometry verification 两段
式验证，这是必须的，不是可选项。
所以你的研究价值，某种程度上不在“高斯画面更好看”，而在于：
长期记忆召回能不能真的降低 long-gap revisit 和 branch confusion。
这也是你实验问题 RQ2 最关键的地方。
第二，archive bank 是否真的“有用”
很多系统做 archive，最后变成“存了一堆东西，但取出来没用”。
你自己报告里也提到了这个风险。
所以 archive 的数据结构不能只是“压缩一下高斯”。它至少要包含：
retrieval descriptor
summary latent
anchor / key Gaussian
reference pose
light geometry summary
metadata
